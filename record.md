# P5 阶段架构能力边界分析

## 当前能力范围评估

架构本身（感知→记忆→规划→执行→学习）是通用的，但**工具层**和**执行模型**是瓶颈，导致能完成的任务范围比较窄。

### 核心限制

| 限制 | 影响 |
|------|------|
| **只有 9 个基础工具** | 本质上就是文件管理 + 网页搜索 + Python 计算器 |
| **execute_python 是隔离沙箱** | 每次调用独立进程，无法跨步骤保持变量/状态，复杂编程任务做不了 |
| **没有 shell 访问** | 不能跑 git、pip、docker、系统命令 |
| **没有浏览器交互** | 不能填表单、点按钮、截图 |
| **没有数据库操作** | 不能直接查 MySQL/PostgreSQL |
| **reply_user 是单向终止** | 执行过程中无法向用户提问再继续（中断即结束） |
| **纯串行执行** | 不能并行调用多个工具 |

### 实际能做的 vs 做不了的

**能做：** 简单问答、读写本地文件、搜索网页信息摘要、独立的 Python 计算、简单的 API 调用

**做不了：** 写一个完整项目、调试代码、部署服务、操作数据库、管理 Git 仓库、多轮交互式任务

## 突破方向（按收益排序）

1. **加 shell 工具** — 能跑任意命令，能力直接翻倍（git、pip、curl、docker 全有了）
2. **有状态的 Python 环境** — 用持久化的 IPython session 替代一次性沙箱，步骤间可以传递变量
3. **执行中交互** — reply_user 不终止流程，等用户回复后继续执行
4. **MCP 协议接入** — 通过 Model Context Protocol 接入外部工具生态（数据库、浏览器、云服务）

其中 1 和 2 改动最小、收益最大。
