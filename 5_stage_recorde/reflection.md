# 架构反思：从错误的拆分到正确的认知

## 走过的路

从 P0 到 P5，我们构建了一个完整的多 Agent 系统：

```
P0: 基础框架（LangGraph 状态机）
P1: 工具接入（9 个工具）
P2: 三层记忆（短期/长期/情景）
P3: 学习闭环（Learner + Prompt 自优化）
P4: 单 Agent 优化（去噪 + 解耦 + 感知升级）
P5: 评估体系（结构指标 + LLM-as-Judge + Benchmark）
```

每个阶段都解决了真实的工程问题，积累了宝贵的实战经验。

## 核心错误：按认知功能拆分 Agent

我们的五个 Agent 本质上是把**一个人的大脑**拆成了五块：

```
Perceiver  = 眼睛（感知环境）
MemoryAgent = 海马体（检索记忆）
Planner    = 前额叶（制定计划）
Executor   = 手（执行动作）
Learner    = 反思能力（提取经验）
```

这五个不是五个人，而是一个人的五种认知功能。把它们拆成独立 Agent 导致了：

1. **信息瓶颈** — Planner 看不到完整对话，Executor 看不到规划推理过程，每个 Agent 都只拿到了残缺的上下文
2. **无意义的开销** — 用户说"你好"，系统调了 5 次 LLM，而一个完整的 Agent 只需要 1 次
3. **经验噪声** — Learner 对简单任务强行挤出"经验"，污染长期记忆
4. **频繁 replan** — Planner 和 Executor 的分离导致计划经常不完整，需要多轮来回

## 正确的拆分方式：按工作职责

多 Agent 架构本身没有问题，问题在于拆分的维度。正确的做法是让每个 Agent 成为一个**完整的人**，拥有独立的感知、规划、执行能力，但扮演不同的**工作角色**：

```
错误：按认知功能拆分（一个人切成五块）
  感知Agent → 记忆Agent → 规划Agent → 执行Agent → 学习Agent

正确：按工作职责拆分（一个团队的不同成员）
  Leader（分解任务、分配、审查结果）
    ├── 研究员（搜索信息、调研方案）
    ├── 开发者（写代码、改文件）
    └── 测试员（运行测试、验证结果）

每个角色 = 独立的 agent loop + 自己的工具集 + 自己的上下文
```

核心区别：**按认知功能拆分是错的，按工作职责拆分才是对的。**

## 错误中学到的

虽然架构方向走错了，但这段经历带来了不可替代的经验：

### 工程层面
- **消息序列污染**：非工具 Agent 返回 AIMessage 会破坏 LangChain 消息格式校验 → 必须返回空 messages
- **空计划崩溃**：Executor 不做越界检查直接索引 → 永远不要信任上游数据
- **状态管理**：LangGraph 的 `add_messages` reducer 如何工作，状态如何在节点间流转
- **Prompt 工程**：大括号转义、JSON 输出格式控制、幻觉检测与过滤

### 架构层面
- **Agent 间通信的代价**：每多一次 Agent 调用就多一次信息压缩和丢失
- **记忆系统的价值与陷阱**：向量检索需要相关性阈值，否则噪声比信号多
- **评估的必要性**：没有评估就不知道优化是否有效，结构指标 + LLM-as-Judge 是实用的组合
- **信任模型**：现代 LLM 足够强大，不需要外部系统帮它"思考"，只需要给它好的工具和知识

### 认知层面
- **拆分的维度决定了架构的成败** — 同样是多 Agent，按认知功能拆和按工作职责拆，效果天壤之别
- **简单的架构 + 强大的工具 > 复杂的架构 + 基础的工具**
- **走过错误的路才知道什么是对的** — 如果一开始就做单 Agent，可能永远不会理解为什么多 Agent 的拆分方式如此重要

## 下一步方向

如果要重新设计，应该走这条路：

1. **单 Agent 主循环**：一个模型，一个 loop，模型自己决定调什么工具、何时停止
2. **按需派生子 Agent**：只在任务确实需要并行/隔离时才拆分（按角色，不按功能）
3. **保留记忆系统**：跨会话记忆是模型不自带的，这是 Harness 的核心价值
4. **大幅扩展工具层**：shell、有状态 Python、MCP 协议接入——工具层是真正的能力瓶颈

---

> 错误本身就是有意义的。意识到了之后，至少它阻止了我们继续向错误的方向发展。
