# P5: Agent 评估框架

## Context
P4 已完成（去噪 + 解耦 + 感知升级），但目前没有任何方式量化各 Agent 的能力。需要建立三层评估体系：**结构化指标 + LLM-as-Judge + Benchmark 测试集**，用于衡量质量并防止退化。

## 目录结构
```
eval/
  __init__.py
  run_eval.py           # 入口: python eval/run_eval.py
  harness.py            # 核心: graph.stream() 捕获每个节点输出
  test_cases.yaml       # 声明式测试用例定义
  metrics/
    __init__.py
    structural.py       # 自动化结构检查
    llm_judge.py        # LLM 评分 (1-5)
  report.py             # 汇总报告生成
  results/              # 运行结果 (gitignore)
```

## 实现步骤

### Step 1: 创建 eval/ 脚手架
- 创建上述目录和 `__init__.py`
- `eval/results/` 加入 `.gitignore`

### Step 2: harness.py — 测试执行器
- 核心思路：用 `graph.stream(state)` 捕获每个节点的 state update，**零侵入**不改 agent 代码
- `graph.stream()` yield `{node_name: state_update}` — LangGraph 原生 API
- execute 节点会循环多次，收集为列表
- 每个 test case 前 `shared_short_term.messages.clear()` 防止跨测试污染
- 关键代码模式：
```python
for event in graph.stream(initial_state):
    for node_name, update in event.items():
        if node_name == "execute":
            node_outputs["execute"].append(update)
        else:
            node_outputs[node_name] = update
```

### Step 3: test_cases.yaml — 声明式测试用例
5 类场景，共约 10 个 case：

| 类别 | 示例 | 期望 |
|------|------|------|
| conversation | "你好" | 0 工具需求, reply_user, 无 replan |
| simple_tool | "读取 /tmp/test.txt" | 包含 read_file, 无错误 |
| multi_step | "找 config.py 并读取 MODEL_NAME" | >=2 步, 回复含 gpt-4o |
| ambiguous | "改一下那个文件" | 用 reply_user 澄清 |
| error_recovery | "读取 /不存在的路径" | 有错误, 但仍有回复 |

每个 case 含：`id`, `category`, `input`, `setup`(可选), `expect`(结构检查), `judge`(LLM评分阈值)

### Step 4: metrics/structural.py — 结构化指标
对每个 Agent 做**格式/规则校验**：

- **Perceiver**: JSON 解析成功、intent 非空、needs_tools 只含合法工具名
- **Planner**: JSON 解析成功、最后一步是 reply_user/task_complete、工具名合法、步数范围
- **Executor**: 错误计数、成功率、是否有/无错误（按用例期望）
- **End-to-end**: 有回复、replan 次数、回复包含关键词

### Step 5: metrics/llm_judge.py — LLM 评分
- 用同一套 OpenAI API (config.py)，temperature=0
- 单次调用评 4 个维度：intent_accuracy, plan_quality, task_fulfillment, response_quality (1-5)
- 可通过 `--no-judge` 跳过（省 API 费用）

### Step 6: report.py — 报告生成
- 按 category 汇总通过率
- 按 case 列出每个指标的 pass/fail
- LLM 评分的平均分
- 输出到终端 + 保存 JSON

### Step 7: run_eval.py — 入口
```bash
python eval/run_eval.py                    # 跑全部
python eval/run_eval.py --runs 3           # 每个 case 跑 3 次（应对非确定性）
python eval/run_eval.py --category simple_tool  # 只跑某类
python eval/run_eval.py --no-judge         # 跳过 LLM 评分
```

## 关键设计决策
1. **graph.stream() 零侵入** — 不修改任何 agent 代码，直接从 LangGraph 流式 API 捕获中间结果
2. **YAML 声明式用例** — 新增用例不写 Python，只加 YAML
3. **每 case 重建图** — 避免跨测试的短期记忆污染
4. **非确定性处理** — 多次运行 + 多数投票；格式检查必须 100% 通过，内容检查用多数投票

## 需要修改的现有文件
- `.gitignore`: 添加 `eval/results/`

## 验证方式
```bash
# 快速验证（无 LLM 评分）
python eval/run_eval.py --no-judge --category conversation

# 完整验证
python eval/run_eval.py --runs 2
```
预期：看到格式化报告，每个 case 的结构指标 pass/fail 和 LLM 评分。
